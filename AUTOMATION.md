# Automation: Dependency Updates, CI, and Releases

This repo follows Micronaut’s official maintenance pattern: Renovate for version updates, CI on every PR, and automatic release tagging when the Micronaut version changes.

## 1) Dependency & Micronaut Version Updates (Renovate)

**Config:** `.github/renovate.json`

What it does:
- Monitors Gradle dependencies and GitHub Actions.
- Schedules updates after **10 PM Europe/Prague**.
- Auto-merges **patch** updates.
- Creates a **Dependency Dashboard** issue for visibility.

Micronaut version handling:
- Renovate updates `io.micronaut.platform:micronaut-parent` in `gradle/libs.versions.toml`.
- The version source of truth is `versions.micronaut-platform` in `gradle/libs.versions.toml`.

If Renovate opens a breaking PR:
- Maintainership reviews and fixes (no auto-merge for minor/major).

## 2) CI (Build + Tests)

**Workflow:** `.github/workflows/gradle.yml`

Triggers:
- Push and PR to `master` and release branches like `4.10.x`.

Steps:
- Setup Java 21 (Temurin)
- Run `./gradlew check`

The CI must pass before merging any Renovate or manual PR.

## 3) Releases (Tag + GitHub Release)

**Workflow:** `.github/workflows/release.yml`

Trigger:
- Push to `master` or `*.x` branches **when `gradle/libs.versions.toml` changes**.

Behavior:
1. Reads `versions.micronaut-platform` from `gradle/libs.versions.toml`.
2. Checks whether it changed versus the previous commit.
3. Computes the tag name: `v<version>`.
4. If the version changed and the tag doesn’t exist, it:
   - creates the git tag
   - pushes it to origin
   - publishes a GitHub Release with autogenerated notes

Example:
- If `versions.micronaut-platform = "4.10.2"`, the workflow creates tag `v4.10.2`.

## 4) Operating the System

**Normal flow:**
1. Renovate opens a PR bumping Micronaut.
2. CI runs and must pass.
3. Merge the PR.
4. The release workflow tags and publishes a GitHub Release automatically.

**If release tag already exists:**
- The workflow detects it and does nothing.

## 5) Ownership & Recovery

- A maintainer must review minor/major Renovate PRs.
- If CI fails, fix the code in the same PR.
- If a tag was created incorrectly, delete the tag + release and re-run.

## 6) Files Summary

- `.github/renovate.json` — dependency updates and scheduling
- `.github/workflows/gradle.yml` — build/test validation
- `.github/workflows/release.yml` — tagging and GitHub Release
